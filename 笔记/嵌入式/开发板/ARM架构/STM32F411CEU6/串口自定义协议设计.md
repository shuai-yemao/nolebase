## 2. 逐点精讲

### 知识点 1：串口自定义协议的设计

#### 实际意义

对于任何与利益相关的数据都是需要保密的，而串口自定义协议就是为了防止数据在串口传递过程中被他人获取并被解析

#### 辅助图示

1. 消息队列串口架构 ![[Pasted image 20260404123223.png]]

#### 核心逻辑/原理

其核心为将 APP 层与 BSP 解耦，只对外提供 API 接口，同时输出的数据要选择加密，通过设定的协议来解包

#### 关键公式/结论

1. 设计自定义协议的基础要求就是需要面对对象的程序设计能力
2. 熟练搭建数据处理架构结合外设的使用是串口设计的重要能力
3. 协议的自定义设计需要考虑到安全和稳定以及高速运行下的可能产生的问题
4. 串口 bsp 层的任务为内存管理，防止数据覆盖和数据丢失
5. 当数据包净荷出现包尾，以及包尾误码如何解决？在数据包中加入数据长度，检测到包尾通过数据长度来判断是否以及传输完毕，包尾误码可以通过数据长度判断本次传输是否已经完成
6. 在传输中也可以加入看门狗来判断本次传输是否完成，按传输字节来设定喂狗时间，超过则出现异常，触发看门狗中断
7. 在实际使用中串口极易受到环境影响而导致误码，如由于数据线过长导致数据准备时间长，波形呈现弧形
8. 如果 MCU 使用的晶振有较大误差，经过分频、PLL 和倍频，以及感性元件后，串口传递的数据很可能是乱码
9. 通过 CRC 校验，哈希值以及密码学来检测数据的准确性

## 4. Q&A

### Q 1：为什么需要协议设计？协议设计解决了什么问题？

 A 1: 协议设计是为了保障串口通信的安全、可靠和高效。主要解决以下问题：

1. **安全性**：防止数据在传输过程中被窃取或篡改，通过加密和自定义格式增加破解难度。
2. **可靠性**：通过校验机制（如 CRC、哈希）检测误码，确保数据完整；定义帧头、帧尾、长度字段，避免粘包、断包。
3. **解耦与维护**：将 APP 层与底层硬件（BSP）分离，提供统一的 API 接口，便于移植和升级。
4. **流量控制**：协调发送与接收速度，防止数据覆盖或丢失（如使用缓冲区、DMA、消息队列）。
5. **错误处理**：针对环境干扰（信号衰减、时钟误差）设计重传、超时机制，提升系统鲁棒性。

### Q 2：当 CPU 处理的 APP 线程读取速度大于 DMA 写入速度，如何处理？

A 2: 此时读取线程会很快消耗完缓冲区数据，然后等待。常用处理方式：

1. **同步机制**：让读取线程在缓冲区空时阻塞（如使用信号量、消息队列），DMA 写入后释放信号量唤醒线程。
2. **调整优先级**：适当降低读取线程优先级，避免不断轮询消耗 CPU 资源。
3. **增大缓冲区**：增加缓冲区大小，让 DMA 能持续写入更多数据，减少读取线程等待频率。
4. **双缓冲区（Ping-Pong Buffer）**：一个缓冲区用于 DMA 写入，另一个用于 CPU 读取，写满后交换指针，实现无锁并发。
5. **流量控制**：若协议支持，可通知发送方降低发送速率（如硬件流控 RTS/CTS）。

### Q 3：当 DMA 写入速度大于 CPU 读取速度，如何处理？

A 3: 此时缓冲区可能溢出，导致数据丢失。解决方案：

1. **增大缓冲区**：扩大缓冲区容量，容纳更多待处理数据。
2. **硬件流控**：启用 RTS/CTS，当缓冲区快满时拉高 RTS 通知发送方暂停。
3. **软件流控**：发送 XOFF/XON 字符控制对端发送。
4. **提高读取线程优先级**：让 CPU 及时处理数据，避免堆积。
5. **DMA 双缓冲/环形缓冲区**：利用 DMA 半传输/传输完成中断，在另一半缓冲区写满前通知 CPU 读取。
6. **超时机制**：若数据积压超过阈值，可丢弃最旧数据（如循环覆盖）或触发错误处理。
7. **优化读取逻辑**：减少 CPU 处理耗时，或使用 DMA 直接传输到内存（如内存到内存），减轻 CPU 负担。

### Q 4：APP 订阅线程优先级低，未处理数据？是否要移动缓冲区指针？移动了，数据会被覆盖吗？

A 4:

- **是否移动指针**：不应在数据未被处理时移动读指针，否则该数据将丢失。
- **数据覆盖风险**：如果写指针追上读指针（缓冲区满），新数据会覆盖未处理的旧数据。
- **解决方案**：
  1. **确保处理完成再移动**：读指针只在 APP 线程成功取出数据后前进。
  2. **增加缓冲区大小**：提供足够空间，容忍处理延迟。
  3. **提升线程优先级**：让 APP 线程及时调度，避免数据积压。
  4. **使用多缓冲区或队列**：每个数据包独立存放，写满后启用备用缓冲区。
  5. **超时丢弃**：若数据长时间未处理，可主动丢弃并移动指针，同时记录错误。
  6. **流控反馈**：缓冲区快满时暂停接收（流控），防止覆盖。

### Q 5：超长数据包不分包传输导致数据被覆盖？

A 5: 超长数据包容易超出缓冲区容量，造成数据覆盖。预防措施：

1. **协议规定最大长度**：定义数据包最大长度（如 256 字节），超过则必须分包。
2. **接收方长度检查**：解析时若发现长度超出缓冲区，可丢弃或请求重传。
3. **动态缓冲区**：根据长度临时分配内存（需注意碎片与实时性）。
4. **流式处理**：边接收边处理，不等待整个包到达（适合大文件传输）。
5. **双缓冲接力**：当第一个缓冲区满时，立即启动第二个缓冲区，由 CPU 在后台处理前一个缓冲区。
6. **分包/重组机制**：发送方将大数据包拆分成小块，接收方按序号重组，确保完整性。
7. **增加缓冲区容量**：根据应用场景预留足够空间。
