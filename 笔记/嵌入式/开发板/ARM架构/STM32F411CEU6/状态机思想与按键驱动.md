### 知识点 2：多种按键控制方式

#### 核心逻辑/原理

按键主要的点就在于消抖和确定扫描时间

#### 关键公式/结论

1. 按键驱动方式有轮询、[[中断]]、状态机、事件驱动
2. 实际企业项目用的是事件驱动 + 状态机框架（可以参考资料链接中 GitHub 的代码库）
3. 在设计按键驱动中，主要的问题是要确保按下时间的准确性
4. 不同的驱动方式计时方式不同，中断可以设计中断时间，在操作系统中任务调度受多方面影响不能使用计数器计时，需要获取系统 tick 来确认时间

#### Typedef 关键字的作用是什么？ # define dps struct s* 和 typedef struct s* tps 哪种方法更好？为什么？Typedef 和 # define 有什么区别

1. Typedef 的作用是为一种数据类型定义一个新名字

### Q 2：当 freertos 以抢占式调度运行时，任务 b（优先级低）正在运行，然后，到达下一个 ostick 时，任务 a（优先级高）就绪了，那这时候 freertos 会从任务 b 切换到任务 a 运行

#### 那切换后的任务 B 的状态是怎样的？

B 任务进入就绪模式，等待 CPU 重新调度，当任务 a 阻塞或者被挂起

#### SVC 和 PendSV 中断的区别是什么？（在）

SVC（系统服务调用，亦简称系统调用）和 PendSV（可悬起系统调用），它们多用于在操作系统之上的软件开发中。

1. SVC 用于产生系统函数的调用请求，用户程序使用 SVC 发出对系统服务函数的呼叫请求，以这种方法调用它们来间接访问硬件，，SVC 异常是必须立即得到响应的
2. PendSV 的典型使用场合是在上下文切换时（在不同任务之间切换）与 SVC 不同，它是可以像普通的中断一样被悬起的（往 NVIC 的 PendSV 悬起寄存器中写 1）

#### 在 PendSV 中，任务究竟是如何切换的？

在 FreeRTOS 中，PendSV 异常处理程序负责实际的上下文切换。当调度器决定切换任务时（例如在 tick 中断中触发），它会设置 PendSV 挂起位，然后退出中断。随后，PendSV 处理程序在较低优先级上下文中执行，安全地进行任务切换。具体步骤：

![[Pasted image 20260317145424.png]]

1. **触发 PendSV**：
    
    - 调度器（如在 xPortSysTickHandler 中）检测到需要任务切换（如更高优先级任务就绪），通过设置 NVIC 的 ICSR 寄存器中的 PENDSVSET 位挂起 PendSV 异常。
2. **进入 PendSV 处理程序**：
    
    - 当处理器退出当前中断或异常后，PendSV 被触发，执行 vPortPendSVHandler（或类似函数）。
3. **保存当前任务上下文**：
    
    - 将当前运行任务（如任务 B）的 CPU 寄存器（如 R0-R3, R12, LR, PC, xPSR）压入其堆栈（任务堆栈）。
    - 更新任务控制块（TCB）中的堆栈指针（pxCurrentTCB->pxTopOfStack）指向保存后的堆栈顶。
4. **选择下一个任务**：
    
    - 调用调度器函数（如 vTaskSwitchContext）从就绪列表中选择最高优先级的任务（如任务 A），并更新 pxCurrentTCB 指向新任务的 TCB。
5. **恢复新任务上下文**：
    
    - 从新任务的 TCB 中获取堆栈指针（pxCurrentTCB->pxTopOfStack）。
    - 从堆栈中弹出寄存器值（恢复 R0-R3, R12, LR, PC, xPSR 等）。
    - 调整堆栈指针以匹配恢复后的状态。
6. **退出 PendSV 并切换到新任务**：
    
    - 通过修改 LR 寄存器（如设置 EXC_RETURN 值）和处理器状态，退出异常模式，返回到新任务（如任务 A）的代码位置继续执行
