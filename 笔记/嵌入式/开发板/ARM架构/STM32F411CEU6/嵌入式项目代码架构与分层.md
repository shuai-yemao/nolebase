## 2. 逐点精讲

### 知识点 1：嵌入式软件架构和分层设计

#### 实际意义

架构和分层的设计让各个层实现高内聚低耦合，让每个层独立，修改一个层的程序不需要去修改另外的层

#### 辅助图示

![[Pasted image 20260311161333.png]]

#### 核心逻辑/原理

嵌入式分层架构的核心目标是**解耦**，即通过明确各层级的职责边界，实现 " 修改某一层逻辑时不影响其他层 "，同时提升代码复用性、可维护性和可移植性。

#### 关键公式/结论

1. Driver 层：CPU 厂商和 MCU 厂商提供的驱动库，用于 CPU 和片上外设编程
	1. ==STM 32 的 HAL（硬件抽象层）位于此层==，提供 CPU 内部 GPIO、[[IIC]]、[[UART]] 等外设底层接口
	2. 直接操作 CPU 的寄存器来实现时钟系统，总线控制以及中断等
	3. 更换芯片需要修改此层代码
	4. ==仅为 Core 层提供硬件操作接口，不与 BSP 层进行通信和交互==
2. Core 层：面向 MCU 编程，用于初始化 MCU 内部外设（[[ADC]]，TIM ,[[UART]] 等），还包含 main. C 文件
	1. 统一外设接口，使上层（如 BSP） 开发无需关注芯片硬件差异
	2. ==对常用外设 [[嵌入式/通信协议/有线通信协议/SPI|SPI]], [[IIC]], [[UART]] 进行高层封装，标准化接口==
3. BSP 层：面向板上外设编程，如 mpu 6050 等
	1. ==针对板上外设（CPU 未内置的）调用 Core 库的代码（不与 Driver 层交互），封装器件具体功能的操作流程==；举例：mpu 6050 读取三维数据，[[IIC]] 初始化，发送指令，接收寄存器数据，数据校验，显示数据
	2. 与具体硬件板卡强相关，同一芯片的不同板卡（如 STM32F103 的开发板 A 和开发板 B）需单独适配 BSP 层
4. OS 层：==提高系统实时性能和系统并行能力==
	1. 通过调度器为系统提供内存资源（flash 和 SRAM 以及外部存储器）管理能力，避免资源冲突
	2. 利用信号量来实现任务间的互斥，同步以及通信
	3. ==同时调用 OS 时要考虑死锁问题和优先级反转问题==
	4. 使用操作系统实现多任务，调用 BSP 层驱动程序，对设备进行操作，并将数据存储在缓冲区或发送给 APP 层。
5. Middlewares 层：主要是项目之间通用的外设抽象而出的程序
	1. 模块化设计，可快速开发
	2. 常用中间件有 LVGL, [[MQTT]] 等
6. System 层：对系统定义整体的宏和对全局系统有影响的参数和文件
7. APP 层：也叫业务逻辑层，结合实际需求具体的应用逻辑，task 文件存放位置
	1. 只调用 OS 层和中间件层来完成逻辑算法，保证代码的独立性
	2. 用户具体要求的代码实现如人机界面交互，微信小程序通过物联网平台与设备通信，机械臂的六维移动
8. ==设计每层代码时，一定要避免跨层操作==
	1. 跨层操作会导致导致逻辑不清晰
	2. 考虑到部分中断，rtos 无法管理，会影响系统运行，数据丢失，任务闲置
9. ==Driver 是对硬件（寄存器）的抽象，Core 是对 MCU（外设）的抽象==

## 4. Q&A

### Q 1：HAL 库是在哪一层被调用的？

 A 1: HAL 库在 Driver 层中和 Core 层中被调用，在底层硬件中承上启下

### Q 2：OS 实现 APP 之间信息传递的方式有哪些？

A 2:

1. 消息队列：队列可以在任务与任务之间以及任务与中断之间的信息传递，任务接收来着其他任务的不定长消息，队列为空时读取队列任务阻塞
2. [[信号量]]：信号量可以实现任务之间的同步和互斥以及临界资源访问，信号量为零，所有访问它的任务都阻塞，互斥量还有优先级继承（防止优先级反转问题）
3. [[事件组]]：事件主要用于多任务间的同步，无数据量的传输，也可以一个任务需要多个条件下才能激活
4. [[任务通知]]：存储在任务的 TCB 结构体中，功能与信号量，队列相似，不过内存开销更小，解除阻塞任务速度更快，但是只能有一个任务接收通知
5. 共享存储区（环形缓冲区，数组，队列   ）：BSP 将数据存储在缓冲区中，任务从共享存储中存取数据，这里要加上互斥锁，防止冲突

### Q 3：OS 实现 APP 和 BSP 之间的信息传递的方式有哪些？

A 3:

1. 消息队列：BSP 层将传感器数据（如 [[MPU6050]] 的加速度数据）写入消息队列，APP 层任务从队列中读取数据进行处理。
2. [[信号量]]：BSP 层完成硬件操作（如设备初始化）后，释放信号量，APP 层任务等待该信号量后再执行后续逻辑（如显示）。
3. 共享内存：BSP 层和 APP 层共享一块内存区域，BSP 层写入数据，APP 层读取数据，通过互斥锁保护共享内存的访问，避免数据冲突。
4. 回调函数：BSP 层注册回调函数，当硬件事件触发（如传感器数据就绪）时，调用回调函数并释放信号量，APP 层通过信号量响应事件并处理数据。

### Q 4：为什么要程序解耦？怎样让程序解耦？

A 4:

1. 程序解耦的必要性：
	1. 关注点（软件系统可能会遇到的多变的部分）分离
	2. 资源管理
	3. 模块自检增强稳定性
	4. 复用性高开发速度快
	5. 模块化方便调试和测试
	6. 设备兼容性强
2. 程序解耦的步骤
	1. 首先要有 C 语言基础（[[指针]]，[[内存管理]]，结构体等）
	2. 其次需要能配置不同外设模块的寄存器实现具体功能并且能与 MCU 互通
	3. 之后程序中要有自检代码，通过返回值来定位问题
	4. 然后实现驱动程序的线程安全，驱动出问题不会影响系统运行，且及时回收系统资源
	5. 最后，根据面向对象思想将代码抽象化出模板和实例化到模块中去在实际中验证代码可行性

### Q 5：OS&Core 连接主要在做什么？

A 5: 答：OS 会和 Core 里的 systick 中断回调产生联系，因为 systick 回调里会触发 pendsvc [[中断]]，产生 OS 的任务调度切换，同时，在任务切换过程中，需要从就绪列表里找到最高优先级的任务，也是 Core 层提供的。用于给 OS 提供调度接口，应用层和底层不产生直接的联系，使用统一的中间接口进行调用，使其完全解耦。

### Q 6：软件架构基础中，CMSIS 是哪一层？

A 6: 属于 driver 层，这是 ARM 芯片公司提供的对 CPU 进行操作的驱动 API
